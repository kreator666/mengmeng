# 量化回测系统开发计划

> 基于 `design/量化回测系统方案.md` 制定
> 版本：v1.0
> 制定日期：2026-07-14

---

## 1. 项目目标

构建一个支持 **现货 + 合约** 双市场的加密货币量化交易回测系统。

- 用户通过 Web 界面输入**因子公式**或 **Python 代码**作为策略信号
- 系统从 **Gate.io API v4** 按需拉取历史 K 线数据
- 执行**向量化回测**
- 输出完整的**绩效分析报告**与**可视化图表**

**适用范围**：不超过 10 人的研究型用户群体，暂不接实盘。

## 2. 技术栈与架构

### 2.1 技术栈

| 层级 | 技术选型 |
|------|---------|
| 前端 | React 18 + TypeScript + Vite |
| UI 组件 | Ant Design / shadcn |
| K 线图表 | TradingView Lightweight Charts |
| 数据图表 | ECharts / Recharts |
| 后端 | FastAPI + Uvicorn + Pydantic v2 |
| 回测引擎 | Pandas + NumPy |
| 因子引擎 | Pandas + NumPy + AST + restrictedpython |
| 数据层 | SQLite + Parquet + aiohttp/httpx |
| 部署 | Docker Compose |

### 2.2 系统分层

```
前端层 (React + TypeScript)
    ↓
API 网关层 (FastAPI)
    ↓
业务服务层 (Pydantic 模型 + 服务编排)
    ↓
回测引擎层 (Pandas/NumPy 向量化回测)
    ↓
因子引擎层 (公式解析 + Python 沙箱执行)
    ↓
数据管理层 (Gate.io 客户端 + 缓存 + SQLite/Parquet)
```

## 4. 详细任务分解

### M1：数据层（第 1 周）

**目标**：建立稳定、高效、可复用的数据获取与缓存能力。

| 编号 | 任务 | 内容 | 交付物 |
|------|------|------|--------|
| M1-1 | Gate.io API 客户端封装 | 封装现货/合约 K 线接口；处理 `from/to/limit` 分页；USDT/BTC 结算合约支持；标记价格/指数价格支持 | `backend/app/data/gateio_client.py` |
| M1-2 | 数据标准化 | 统一现货与合约返回字段顺序差异，转换为标准 OHLCV DataFrame | 数据转换模块 |
| M1-3 | 缓存管理器 | 缓存键：`{symbol}_{interval}_{market_type}`；实现完全命中/部分缺失/完全未命中的按需拉取策略；缺失区间智能合并 | `backend/app/data/cache_manager.py` |
| M1-4 | 数据存储 | SQLite（< 100 万条）+ Parquet（≥ 100 万条）双格式；时间范围查询优化 | `backend/app/data/data_store.py` |
| M1-5 | 限流与重试 | 200ms 请求间隔；指数退避重试；错误日志记录 | 限流模块 |
| M1-6 | 数据 API | `/api/symbols` 获取交易对列表；`/api/data/klines` 获取 K 线数据 | `backend/app/api/data.py` |

**验收标准**：
- 首次拉取 BTC_USDT 1h 数据（1 年）成功
- 重复请求同一区间零 API 调用
- 部分缺失区间仅补充缺失部分

---

### M2：回测引擎（第 2 周）

**目标**：实现向量化回测核心，支持多种仓位与成本模型。

| 编号 | 任务 | 内容 | 交付物 |
|------|------|------|--------|
| M2-1 | 回测核心 | 收益率计算、信号 `shift(1)` 避免未来函数、策略收益、交易成本扣除、资金曲线生成 | `backend/app/core/backtest_engine.py` |
| M2-2 | 仓位管理 | 固定金额、固定比例、凯利公式、波动率目标（ATR） | `backend/app/core/position_manager.py` |
| M2-3 | 成本模型 | 现货/合约 Maker/Taker 手续费、滑点、合约资金费率 | `backend/app/core/cost_model.py` |
| M2-4 | 交易记录生成 | 根据仓位变化生成 entry/exit 交易明细 | 交易记录模块 |
| M2-5 | 回测 API | `POST /api/backtest/run` 接收策略配置并返回回测 ID | `backend/app/api/backtest.py` |

**验收标准**：
- 输入简单 MA 金叉信号，输出资金曲线与交易记录
- 10000 根 K 线回测耗时 < 1 秒

---

### M3：因子引擎（第 3 周）

**目标**：支持公式和 Python 代码两种因子输入模式，实现多因子组合与 IC 分析。

| 编号 | 任务 | 内容 | 交付物 |
|------|------|------|--------|
| M3-1 | 内置因子库 | MA、EMA、WMA、MACD、ADX、RSI、MOM、ROC、STOCH、CCI、ATR、BOLL、STD、OBV、VWAP、VOL_MA、ZSCORE、PERCENTILE 等 | 因子库模块 |
| M3-2 | 公式解析 | AST 解析因子表达式，如 `MA(close,10) > MA(close,20) AND RSI(close,14) < 70` | `backend/app/core/factor_engine.py` |
| M3-3 | Python 沙箱 | restrictedpython 执行用户自定义 `factor(df)`；限制 import、文件/网络访问 | 沙箱执行模块 |
| M3-4 | 多因子组合 | 等权组合、加权组合、IC 加权组合 | 组合模块 |
| M3-5 | IC 分析 | Spearman 秩相关系数计算；IC 时间序列；因子相关性检测 | IC 分析模块 |
| M3-6 | 因子 API | `/api/factor/eval` 预览因子信号；`/api/factor/builtins` 获取内置因子列表 | `backend/app/api/factor.py` |

**验收标准**：
- 公式模式支持所有内置因子
- 用户 Python 代码在沙箱中安全执行
- IC 分析输出正确，IC > 0.05 判定为有效因子

---

### M4：绩效分析（第 4 周前半周）

**目标**：输出专业、完整的绩效评估报告。

| 编号 | 任务 | 内容 | 交付物 |
|------|------|------|--------|
| M4-1 | 收益类指标 | 总收益率、年化收益率、超额收益（Alpha） | `backend/app/core/performance.py` |
| M4-2 | 风险类指标 | 最大回撤、最大回撤持续期、波动率、下行波动率 | 风险指标模块 |
| M4-3 | 风险调整收益 | 夏普比率、索提诺比率、Calmar 比率、Sortino 比率 | 风险调整模块 |
| M4-4 | 交易统计 | 总交易次数、胜率、盈亏比、期望收益、最长连胜/连败 | 交易统计模块 |
| M4-5 | 报告生成 | 标准化 JSON：summary / equity_curve / trades / drawdown_series | 报告生成模块 |
| M4-6 | 结果 API | `/api/backtest/{id}`、`/api/backtest/{id}/equity`、`/api/backtest/{id}/trades` | 结果 API |

**验收标准**：
- 输出 ≥ 20 项指标
- 资金曲线、回撤序列、交易记录数据完整准确

## 8. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 未来函数 | 回测结果虚高 | 引擎强制 `signal.shift(1)`；文档明确说明 |
| 代码安全 | 用户 Python 代码执行风险 | restrictedpython 沙箱；禁止 import、文件/网络访问 |
| API 限流 | 数据拉取失败 | 200ms 请求间隔；指数退避重试；失败告警 |
| 数据精度 | K 线为聚合值，无 Tick 数据 | 明确适用日/4h/1h 中低频策略 |
| 合约资金费率 | 回测中难以精确估算 | 提供参数让用户自定义；文档说明近似处理 |
| 并发性能 | < 10 人使用，未做高并发优化 | 异步处理 + 简单队列；必要时限流 |

## 11. 总结

本开发计划将量化回测系统拆分为 **6 个阶段、约 6 周** 完成，核心思路是：

1. **先数据、后引擎、再前端**：确保回测核心逻辑可靠，再构建界面
2. **快速验证、持续集成**：每个阶段结束都有明确可验证的交付物
3. **风险前置**：未来函数、代码安全、API 限流在设计初期即纳入考虑

按此计划执行，可在 6 周内交付一个功能完整、可部署、可扩展的加密货币量化回测系统。
